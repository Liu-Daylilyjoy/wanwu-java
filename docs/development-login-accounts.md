# Development Login Accounts

Date: 2026-07-05

The Docker development environment has two built-in IAM accounts. They are seed accounts plus a MySQL-backed development compatibility repository, not the final reproduced IAM user/role relational model.

| Username | Token | Permission scope | Intended check |
| --- | --- | --- | --- |
| `admin` | `dev-token` | `permission`, `permission.user`, `permission.org`, `permission.role`, `setting`, `model`, `model.model_management`, `app`, `app.rag`, `app.workflow`, `app.agent`, `api_key`, `api_key.api_key_management`, `resource`, `resource.knowledge`, `resource.tool`, `resource.mcp`, `resource.prompt`, `resource.skill`, `resource.safety`, `operation`, `operation.oauth`, `operation.statistic_client`, `exploration`, `exploration.app`, `exploration.mcp`, `exploration.template`, `exploration.skill`, `app_observability`, `app_observability.statistic` | Verify permission-management read views, platform settings, Operation Management, Exploration Square, App Observation, Model Management, Knowledge, Tool, MCP, Prompt, Skill, Safety Guard, RAG, Workflow/Chatflow, Agent, and API Key menus without exposing unreproduced or out-of-scope modules. |
| `app` | `dev-token-app` | `app`, `app.rag`, `app.workflow`, and `app.agent` only | Verify the application-development-only view, including Workflow and Chatflow tabs behind the original `app.workflow` frontend permission. |

Initial password and captcha remain development-compatible so existing zero-change frontend smoke tests still work: use a non-empty placeholder such as `x` or `encrypted`, and captcha code `1234`. After `/user/api/v1/user/password` or `/user/api/v1/user/admin/password` sets a new password, IAM persists a password hash and later logins for that account must use the new password.

`wga` and `wga.wanwu_bot` are intentionally not granted. The Java WGA controllers remain as compatibility shells, but the unchanged frontend hides the General Agent/WanwuBot menu through the IAM permission response.

The Java BFF now also enforces these development permission scopes for `/user/api/v1/**` routes. `dev-token-app` can access the app workspace route families, but direct requests to model, knowledge, resource, permission, operation, setting, exploration, statistic, and API-key management routes return `403` with `permission denied`. Common bootstrap routes such as `/user/api/v1/user/permission`, `/user/api/v1/user/info`, `/user/api/v1/org/select`, avatar/file helpers, and Doc Center routes remain available after login.

Email register/reset/login routes are also available as development compatibility contracts:

- `POST /user/api/v1/base/login/email` returns `dev-token` for `admin` and `dev-token-app` for `app`.
- `POST /user/api/v1/user/login` and `PUT /user/api/v1/user/login` return the same login-session shape as the built-in accounts, including organization, language, and permission data.
- `POST /user/api/v1/base/register/email/code`, `/base/password/email/code`, and `/user/login/email/code` issue the Docker development email code `123456` with a ten-minute TTL; register/reset/two-stage-login endpoints validate and consume that code once. They do not send SMTP email yet. When Java IAM is available, email registration creates an app-scoped user, password reset persists the new password hash, and first-login password change calls the IAM password-change path.
