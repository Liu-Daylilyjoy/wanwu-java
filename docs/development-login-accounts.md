# Development Login Accounts

Date: 2026-07-01

The Docker development environment has two built-in IAM accounts. They are seed accounts plus a MySQL-backed development compatibility repository, not the final reproduced IAM user/role relational model.

| Username | Token | Permission scope | Intended check |
| --- | --- | --- | --- |
| `admin` | `dev-token` | `permission`, `permission.user`, `permission.org`, `permission.role`, `setting`, `model`, `model.model_management`, `app`, `app.rag`, `app.workflow`, `app.agent`, `api_key`, `api_key.api_key_management`, `resource`, `resource.knowledge`, `resource.tool`, `resource.mcp`, `resource.prompt`, `resource.skill`, `resource.safety`, `operation`, `operation.oauth`, `operation.statistic_client`, `exploration`, `exploration.app`, `exploration.mcp`, `exploration.template`, `exploration.skill`, `app_observability`, `app_observability.statistic`, `wga`, `wga.wanwu_bot` | Verify permission-management read views, platform settings, Operation Management, Exploration Square, App Observation, Model Management, Knowledge, Tool, MCP, Prompt, Skill, Safety Guard, RAG, Workflow/Chatflow, Agent, WanwuBot/WGA, and API Key menus without exposing unreproduced modules. |
| `app` | `dev-token-app` | `app`, `app.rag`, `app.workflow`, and `app.agent` only | Verify the application-development-only view, including Workflow and Chatflow tabs behind the original `app.workflow` frontend permission. |

Password and captcha are development placeholders. Use any non-empty password, such as `x`, and captcha code `1234`.

Email register/reset/login routes are also available as development compatibility contracts:

- `POST /user/api/v1/base/login/email` returns `dev-token` for `admin` and `dev-token-app` for `app`.
- `POST /user/api/v1/user/login` and `PUT /user/api/v1/user/login` return the same login-session shape as the built-in accounts, including organization, language, and permission data.
- `POST /user/api/v1/base/register/email/code`, `/base/register/email`, `/base/password/email/code`, `/base/password/email`, and `/user/login/email/code` return deterministic success shells. They do not send email, verify codes, or persist password/user changes yet.

Frontend permissions are intentionally narrower than `web/src/router/constants.js` while the Java reproduction is still incomplete.
