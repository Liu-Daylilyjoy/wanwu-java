# 自主复现工作协议

## 目标

后续 Wanwu Java 复现进入自主推进模式：用户只验收最终可运行版本，开发过程中的规划、源码考证、实现、测试、文档和提交由 Codex 自行完成。

## 信息来源优先级

1. 原 Go 项目：`D:\work\week3\wanwu`
2. 当前 Java 项目：`D:\work\week3\wanwu-java`
3. 前端源码：`web/src`
4. 原项目 Swagger/OpenAPI 文档：`D:\work\week3\wanwu\docs`
5. 本地运行结果、数据库表结构、容器日志

除非缺少本地源码、缺少外部凭据、或需要用户决定业务取舍，否则不向用户提问。

## 开发节奏

- 自行从 Go 源码识别下一个最有价值的前端可访问闭环。
- 每个闭环都按垂直切片实现：API DTO、BFF 适配、应用服务、持久化、Flyway、测试、Docker/HTTP/前端验收。
- 每写完一个可验证小步就提交，提交信息描述业务能力，不提交半成品。
- 每个闭环必须补一份技术要点文档，记录 Go 源码依据、Java 落地结构、数据表、接口契约和验收命令。
- 前端默认零改动；只有证明原前端无法兼容时才考虑修改，并在文档中说明原因。

## 验收门槛

每个闭环提交前至少完成：

- Docker Maven 相关模块测试。
- `docker compose --profile full config`。
- 受影响服务容器启动或健康检查。
- HTTP 级业务闭环验证。
- 涉及前端页面时，用 Playwright 验证真实页面能访问、保存或回显。

## 提交规则

- 保持小提交，每个提交对应一个可解释的能力增量。
- 不把临时截图、运行输出、`.env`、`node_modules`、`target` 提交进仓库。
- 文档与代码同提交，避免只有实现没有复现依据。
- 推送到 `main` 后再向用户汇报最终状态。

## 当前推荐推进队列

1. 智能体删除与列表一致性。
2. 智能体会话草稿列表、创建、清空、删除占位闭环。
3. 智能体发布/版本列表最小闭环。
4. 模型选择器与知识库选择器从空占位升级为 MySQL backed 的可选项。
5. RAG 应用创建、草稿、列表闭环。
